## PRD-1. CLINICAL AI CHAT SUMMARY (NEW FEATURE)

### 1.1 Problem Statement

When a patient has an extended AI companion conversation, the clinician currently has **zero visibility** into what was discussed. The `ChatSession`/`ChatMessage` models exist in Prisma but no endpoint exposes them. Clinicians need a **structured clinical summary** with evidence-backed recommendations, not a wall of conversation text.

### 1.2 Design Principles

1. **Never diagnoses.** The AI provides treatment personalization suggestions and clinical observations only. All language uses "consider", "may benefit from", "literature suggests" — never "patient has" or "diagnosis is".
2. **Cites published evidence.** Each recommendation references specific clinical guidelines or publications (DSM-5-TR, APA Practice Guidelines, NICE guidelines, Cochrane reviews, named CBT/DBT protocols).
3. **Transparent reasoning.** Every recommendation includes the reasoning chain: what the patient said → what clinical pattern it maps to → what the evidence says → what the suggestion is.
4. **Human-in-the-loop mandatory.** Summaries are DRAFT status. No summary reaches the patient or triggers treatment changes without clinician review and explicit approval.
5. **Separation of concerns.** Patient-facing chat remains warm and supportive. Clinical summary is a separate clinician-only artifact generated asynchronously.

### 1.3 Data Model — `ChatSessionSummary`

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | PK |
| `sessionId` | String (FK → ChatSession) | The chat session being summarized |
| `patientId` | String (FK → Patient) | Denormalized for query efficiency |
| `clinicianSummary` | String (encrypted PHI) | Structured clinical summary text |
| `recommendations` | Json | Array of `{title, description, reasoning, evidenceCitations[], signalBand, category}` |
| `evidenceLog` | Json | Array of `{patientStatement, clinicalPattern, citedPublication, publicationYear, evidenceGrade, relevantExcerpt}` |
| `patternFlags` | Json | Array of `{pattern, frequency, severity, suggestedIntervention}` |
| `riskIndicators` | Json | Array of `{indicator, contextQuote, signalBand}` |
| `unknowns` | Json | What the AI could NOT determine |
| `modelVersion` | String | Claude model used |
| `tokenUsage` | Json | Input/output tokens for audit |
| `status` | `DraftStatus` | DRAFT → REVIEWED → APPROVED → REJECTED |
| `reviewedById` | String? (FK → User) | Clinician who reviewed |
| `reviewedAt` | DateTime? | |
| `reviewNotes` | String? | Clinician's notes on the summary |
| `createdAt` | DateTime | |

### 1.4 AI Prompt Architecture

System prompt structure:
1. **Role**: "You are a clinical documentation assistant for licensed behavioral health clinicians. You do NOT diagnose."
2. **Output schema**: Return JSON matching the `recommendations[]` and `evidenceLog[]` structures.
3. **Citation requirements**: APA Practice Guidelines, NICE guidelines, DSM-5-TR, named protocols (CBT, DBT, ACT, MI), Cochrane Reviews, peer-reviewed publications.
4. **Evidence grading**: `A` (RCT/meta-analysis), `B` (cohort/case-control), `C` (expert consensus), `N` (clinical observation).
5. **Guardrails**: Never state a diagnosis. Never recommend medication by name. Always state uncertainty. Flag crisis indicators with direct quotes.
6. **Input context**: Full chat transcript + approved memories + recent submission signal bands + active treatment plan goals.

### 1.5 API Endpoints

| Method | Path | Role Required | Description |
|--------|------|---------------|-------------|
| `GET` | `/clinician/patients/:id/chat-sessions` | CLINICIAN, SUPERVISOR | List chat sessions |
| `GET` | `/clinician/patients/:id/chat-sessions/:sessionId` | CLINICIAN, SUPERVISOR | Full transcript |
| `POST` | `/clinician/patients/:id/chat-sessions/:sessionId/summarize` | CLINICIAN, SUPERVISOR | Generate Clinical AI Summary |
| `GET` | `/clinician/patients/:id/chat-summaries` | CLINICIAN, SUPERVISOR | List summaries |
| `GET` | `/clinician/patients/:id/chat-summaries/:summaryId` | CLINICIAN, SUPERVISOR | Full summary detail |
| `PATCH` | `/clinician/patients/:id/chat-summaries/:summaryId` | CLINICIAN, SUPERVISOR | Review/approve/reject |

### 1.6 Auto-Generation Triggers

- **On session close**: ≥5 patient messages + 30min inactive → auto-generate.
- **On demand**: Clinician clicks "Generate Summary".
- **On escalation**: Crisis detection → immediate summary with riskIndicators prioritized.

## PRD-8. DECISIONS LOG

| Decision | Rationale |
|----------|-----------|
| Patient self-registration blocked at API | Prevents orphaned patients with no org/clinician |
| Voice upload disabled for MVP | Broken stub → ship "Coming Soon" |
| Clinical AI Summary is DRAFT-only | No AI output reaches patients without clinician review |
| Citations reference guidelines, not raw papers | APA/NICE/Cochrane more trustworthy; grade system for transparency |
| AI never diagnoses | "presentation suggests", "consider", "literature indicates" only |
| Chat transcripts read-only for clinicians | Data integrity and patient trust |
| Backup codes Redis → Postgres | Redis eviction/restart could lock users out |
| Per-user rate limits (not just IP) | Clinic NAT shares IP |

### Red Team Hardening Plan

#### RT-1: Authentication Attack Surface

| Test ID | Attack Vector | Test Method | Expected Defense |
|---------|--------------|-------------|-----------------|
| RT-1.1 | Brute force login | Automated 1000-attempt attack on single account | Account locked after 5 attempts, subsequent attempts return 429 |
| RT-1.2 | Credential stuffing | Bulk login attempts with known-breached credentials | Rate limiting + account lockout + IP-based throttling |
| RT-1.3 | Session hijacking | Steal JWT from one browser, use in another | Token bound to user agent + IP (warning on mismatch) |
| RT-1.4 | Session fixation | Pre-set session ID before authentication | New session ID generated on every login |
| RT-1.5 | MFA bypass | Attempt API calls without completing MFA | API rejects tokens without MFA claim |
| RT-1.6 | Password reset abuse | Rapid password reset requests | Rate limited to 3 reset emails per hour |
| RT-1.7 | Invite token replay | Reuse expired/consumed invite token | Token single-use, expires after 72 hours |

#### RT-2: Authorization & Tenant Isolation

| Test ID | Attack Vector | Test Method | Expected Defense |
|---------|--------------|-------------|-----------------|
| RT-2.1 | IDOR on patient data | Clinician requests Patient B's data (not in caseload) | 403 Forbidden — clinician only sees assigned patients |
| RT-2.2 | Tenant data leakage | User A (Tenant 1) requests data with Tenant 2 ID | tenantId enforced on every query — returns empty set |
| RT-2.3 | Role escalation | Patient modifies JWT role claim to CLINICIAN | Server validates role against database, not JWT claim alone |
| RT-2.4 | Horizontal privilege | Patient A accesses Patient B's journal | API enforces `patientId = req.user.id` on all patient data endpoints |
| RT-2.5 | Admin impersonation | Non-admin calls admin-only endpoints | RBAC middleware returns 403 |
| RT-2.6 | Supervisor bypass | Clinician calls supervisor-only escalation endpoints | Role check returns 403 |
| RT-2.7 | 42 CFR Part 2 breach | Export includes substance use data without specific consent | cfr42-guard middleware blocks export of gated fields |

#### RT-3: Data Exfiltration & PHI Exposure

| Test ID | Attack Vector | Test Method | Expected Defense |
|---------|--------------|-------------|-----------------|
| RT-3.1 | Bulk data extraction | Rapid sequential GET requests for all patients | Rate limiting on data-heavy endpoints (10 req/min) |
| RT-3.2 | PHI in error messages | Trigger API errors and inspect responses | Error handler strips PHI from all 4xx/5xx responses |
| RT-3.3 | PHI in logs | Inspect application logs for unencrypted PHI | Logger middleware redacts PHI fields before writing |
| RT-3.4 | PHI in URLs | Check if PHI appears in query parameters/path | PHI only in request body (POST/PATCH), never in URL |
| RT-3.5 | Database direct access | Attempt direct DB connection from compromised app server | Database accepts connections only from API server security group |
| RT-3.6 | Encryption key exposure | Search codebase + environment for hardcoded keys | All keys in AWS KMS / environment variables, never in code |
| RT-3.7 | AI model PHI leakage | Inspect AI prompts for PHI passed to Claude | sanitizeForAI() strips names, DOBs, SSNs before prompt construction |

#### RT-4: Input Validation & Injection

| Test ID | Attack Vector | Test Method | Expected Defense |
|---------|--------------|-------------|-----------------|
| RT-4.1 | XSS in journal entry | Submit `<script>alert('xss')</script>` as journal content | React DOM escaping + CSP blocks inline scripts |
| RT-4.2 | SQL injection | Submit `'; DROP TABLE patients;--` in search | Prisma parameterized queries prevent injection |
| RT-4.3 | NoSQL injection | Submit `{"$gt": ""}` in JSON body | Input validation middleware rejects unexpected operators |
| RT-4.4 | SSRF via AI chat | Submit URL in chat that triggers server-side fetch | AI prompt construction does not follow URLs |
| RT-4.5 | File upload abuse | Upload oversized file or executable as voice memo | Voice upload disabled (501). When re-enabled: file type whitelist + size limit |
| RT-4.6 | Path traversal | Submit `../../etc/passwd` in file paths | Path sanitization in all file-handling routes |
| RT-4.7 | Prompt injection | Submit AI manipulation text in journal/chat | AI system prompt includes injection defense instructions |

#### RT-5: Availability & Resilience

| Test ID | Attack Vector | Test Method | Expected Defense |
|---------|--------------|-------------|-----------------|
| RT-5.1 | DoS via large payloads | Send 100MB request body | Body size limiter returns 413 |
| RT-5.2 | DoS via rapid requests | 1000 requests/second from single IP | Rate limiter throttles at configured threshold |
| RT-5.3 | Resource exhaustion | Open 10,000 WebSocket/SSE connections | Connection limit per IP, idle timeout (60s) |
| RT-5.4 | Cache poisoning | Manipulate cached safety plan data | Cache keyed by user ID + version hash, integrity verified |
| RT-5.5 | Dependency confusion | Supply malicious npm package with same name | Package lock file pins all dependencies + integrity hashes |

#### CRIT-001: PHI Encryption Silent Failure to Plaintext

- **File:** `packages/api/src/middleware/phi-encryption.ts:77-82`
- **Evidence:** The `catch` block at lines 78–82 logs `'PHI encryption failed — field left as plaintext'` but does NOT throw or abort. The plaintext value is written to the database. This is a **HIPAA violation**.
- **Fix:** Replace log-and-continue with `throw new AppError('PHI encryption failed', 500)`.

#### CRIT-002: In-Memory MFA Code Storage

- **File:** `packages/api/src/routes/auth.ts:55`
- **Evidence:** `const pendingMFA: Record<string, string> = {};` — plain object in process memory. No TTL (codes valid forever), no tenant isolation (userId-only key), lost on restart, not shared across ECS instances.
- **Fix:** Use Redis with key `mfa:${tenantId}:${userId}` and 300s TTL.

#### CRIT-003: In-Memory Token Revocation

- **File:** `packages/api/src/routes/auth.ts:58`
- **Evidence:** `const invalidatedTokens = new Set<string>();` — revoked tokens only tracked in this process. After restart or on different instance, revoked tokens are accepted.
- **Fix:** Use Redis `SET` with TTL matching refresh token lifetime.

#### CRIT-004: Cross-Tenant Login

- **File:** `packages/api/src/routes/auth.ts:186`
- **Evidence:** `prisma.user.findFirst({ where: { email: body.email } })` — no `tenantId` in the where clause. If two tenants share an email, wrong user authenticates.
- **Fix:** Require tenant slug/ID in login request, add to where clause.

#### CRIT-005: AI Routes Missing Patient Access Check

- **File:** `packages/api/src/routes/ai.ts:51-55`
- **Evidence:** `prisma.patient.findUnique({ where: { id: body.patientId } })` — verifies patient exists but does NOT check that the authenticated user has caseload access or is in the same tenant. Any authenticated user can access any patient's AI chat/summaries.
- **Fix:** Add `requireCaseloadAccess(req.user!.sub, req.user!.tid, body.patientId)`.

#### CRIT-006: Audit Hash Chain In-Memory with Race Condition

- **File:** `packages/api/src/middleware/audit.ts:13, 139-140`
- **Evidence:** Line 13: `let lastHash = '0'.repeat(64);` — module-scope variable. Lines 139–140: `const hash = hashChain(entryForHash); lastHash = hash;` — no lock, no atomic operation. Concurrent requests can read the same `lastHash` and produce duplicate hashes, breaking chain integrity.
- **Fix:** Read latest hash from DB within a serializable transaction per entry.

#### CRIT-007: Notifications Entirely Stubbed

- **File:** `packages/api/src/services/notification.ts:28-34, 36-39`
- **Evidence:** Lines 28–34: `if (!isProduction)` returns after logging `[STUB]`. Lines 36–39: production SES code is commented out (`// const ses = new SESClient`). The function logs `'Email sent via SES'` at line 39 even though no email was sent — a false positive log. Same pattern for SMS (lines 55–63) and push (lines 81–88). The escalation cascade (lines 100–137) calls these stubs, so T2/T3 escalations never reach anyone.
- **Fix:** Implement SES, Twilio, FCM integrations with delivery confirmation and retry logic.
